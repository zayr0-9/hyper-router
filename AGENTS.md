# hyper-router Agent Guide

This guide is for AI coding agents and human contributors working inside the `hyper-router` repository or building harnesses on top of it. Read this before changing the SDK or integrating it into an external app such as Qubit.

## Project purpose

`hyper-router` is a minimal Node-first TypeScript agent runtime SDK. It provides a reusable core for building agent shells without baking in any specific UI, CLI, TUI, web app, or product workflow.

The SDK owns:

- agent definitions and system instructions
- provider abstraction and provider response normalization
- a transcript-first runtime loop
- local tool execution
- host-mediated tool permission hooks
- transcript and run persistence through storage adapters

The SDK must **not** own:

- terminal UI, web UI, modal rendering, or keybindings
- product-specific slash commands
- app-specific session picker UX
- Qubit-specific JSON-lines protocol shapes
- credentials/secrets management beyond provider option/env lookups

External harnesses such as Qubit should import `hyper-router`, configure providers/storage/tools, and bridge runtime hooks into their own UI/protocol.

## Core runtime model

Main file: `src/core/runtime.ts`.

`AgentRuntime.run(...)` is transcript-first:

1. Load previous transcript messages from storage unless `ephemeral` is true.
2. Filter out stored `system` messages; system messages are runtime-only and regenerated each run.
3. Load prior session metadata if the storage adapter supports it.
4. Build the new input messages from `agent.buildMessages(input)` or a default user message.
5. Prepend current agent instructions as a `system` message.
6. Loop up to `maxSteps`:
   - call `provider.generate(...)`
   - normalize/canonicalize tool call IDs
   - append assistant message and generated images
   - stop if no tool calls
   - resolve permissions for each tool call
   - execute allowed local tools
   - append linked `role: "tool"` result messages
7. Persist transcript/run state/metadata unless `ephemeral` is true.
8. Return a `RuntimeResult` with status, messages, stop reasons, and images.

### Important transcript invariants

Preserve these invariants when editing runtime, storage, or providers:

- System messages are not persisted.
- Assistant messages with tool calls must preserve `toolCalls` order.
- Tool result messages must preserve `toolCallId` linkage.
- Tool result `content` is a JSON stringified `ToolResult`.
- Provider replay should receive the stored transcript without unnecessary mutation.
- Missing tool call IDs are canonicalized as:
  ```txt
  ${toolName}-${step}-${index}
  ```
- Denied tool permissions are represented as normal failed tool results, not thrown runtime errors.

## Defining agents

Main file: `src/core/agent.ts`.

```ts
const agent = defineAgent({
  name: "coding-agent",
  instructions: "You are a concise coding assistant.",
  model: "glm-4.6",
  tools: [readFileTool, writeFileTool],
});
```

Use `buildMessages` only when the host needs custom message construction:

```ts
const agent = defineAgent({
  name: "resume-agent",
  instructions: "Continue from transcript.",
  model: "stub-model",
  buildMessages(input) {
    return input.trim() === ""
      ? []
      : [{ role: "user", content: input, date: new Date() }];
  },
});
```

This is useful for resume flows where the host wants to continue the tool chain without adding an empty user turn.

## Defining tools

Main file: `src/core/tool.ts`.

```ts
const echoTool = defineTool<{ text: string }, { echoed: string }>({
  name: "echo",
  description: "Echo text back.",
  inputSchema: z.object({ text: z.string() }),
  async execute(args, context) {
    return {
      ok: true,
      output: { echoed: args.text },
    };
  },
});
```

Tool requirements:

- `name` should be stable and provider-safe.
- `description` should be concise and action-oriented.
- `inputSchema` can be Zod or JSON Schema depending on provider support.
- `execute` must return `{ ok: true, output }` or `{ ok: false, error }`.
- Thrown tool errors are caught by runtime and converted to failed tool results.
- Tool execution receives `{ sessionId, step }` as context.

## Tool permission system

The permission system is designed for external harnesses. `hyper-router` only emits semantic permission requests and awaits decisions. It does not render modals or prompt users directly.

### Tool-level policies

```ts
const writeFileTool = defineTool<{ path: string; content: string }, { path: string }>({
  name: "write_file",
  description: "Write text to a local file.",
  permission: {
    mode: "ask",
    reason: "Writes to the filesystem.",
    metadata: {
      category: "filesystem",
      severity: "high",
    },
  },
  async execute(args) {
    return { ok: true, output: { path: args.path } };
  },
});
```

Supported modes:

```ts
type ToolPermissionMode = "always" | "ask" | "never";
```

Semantics:

- omitted permission: allow by default, preserving backward compatibility
- `always`: execute without asking
- `ask`: call `hooks.requestToolPermission(request)` before execution
- `never`: deny without asking or executing

If a tool explicitly asks for permission and no hook is configured, runtime denies safely with a failed tool result.

### Runtime default policy

Harnesses can force a default policy for tools without per-tool permission metadata:

```ts
const runtime = createRuntime({
  agent,
  provider,
  storage,
  toolPermission: {
    defaultMode: "ask",
  },
  hooks: {
    requestToolPermission: async (request) => bridgeToHarness(request),
  },
});
```

Resolution order:

1. tool-specific `permission.mode`
2. runtime `toolPermission.defaultMode`
3. implicit `always`

This lets Qubit or another host put all tools behind a permission UI without modifying every tool definition.

### Permission request shape

Defined in `src/core/types.ts`.

```ts
interface ToolPermissionRequest {
  id: string;
  sessionId: string;
  step: number;
  toolCallId: string;
  toolName: string;
  args: unknown;
  description?: string;
  inputSchema?: unknown;
  metadata?: Record<string, unknown>;
}
```

Decision shape:

```ts
type ToolPermissionDecision =
  | { type: "allow" }
  | { type: "deny"; reason?: string };
```

### Harness bridge example

```ts
const pending = new Map<string, (decision: ToolPermissionDecision) => void>();

const runtime = createRuntime({
  agent,
  provider,
  storage,
  toolPermission: { defaultMode: "ask" },
  hooks: {
    async requestToolPermission(request) {
      writeToHarness({
        type: "tool.permission.request",
        id: request.id,
        sessionId: request.sessionId,
        step: request.step,
        toolCallId: request.toolCallId,
        toolName: request.toolName,
        args: request.args,
        description: request.description,
        metadata: request.metadata,
      });

      return await new Promise<ToolPermissionDecision>((resolve) => {
        pending.set(request.id, resolve);
      });
    },
  },
});

function handleHarnessMessage(message: unknown) {
  if (!isPermissionResponse(message)) return;

  const resolve = pending.get(message.id);
  if (!resolve) return;
  pending.delete(message.id);

  resolve(
    message.allow
      ? { type: "allow" }
      : { type: "deny", reason: message.reason ?? "Denied by user." },
  );
}
```

Important harness guidance:

- Do not queue permission responses behind the blocked `runtime.run(...)` call. If the runtime is awaiting a permission hook, responses must be routed to the pending promise immediately.
- Redact or summarize sensitive `args` before displaying if necessary.
- Treat permission requests as semantic events; UI labels/buttons belong in the harness.
- Denials are not exceptions. They become failed tool results visible to the model on the next step.

## Provider responsibilities

Providers implement `ModelProvider.generate(...)` and must normalize provider-specific responses into `ModelResponse`.

Providers should:

- convert SDK messages into provider input format
- convert SDK tool definitions into provider tool schemas
- ask the model to produce tool calls, not execute tools remotely
- normalize tool calls into SDK `ToolCall[]`
- preserve assistant text, tool calls, reasoning, stop reasons, and generated images when available

Providers should not:

- execute local tools
- ask permission for local tools
- persist transcripts
- contain harness-specific behavior

### OpenRouter provider

- Uses `@openrouter/agent`.
- Converts messages through `providers/openrouter/items.ts`.
- Uses OpenRouter tools with `execute: false`; runtime executes tools locally.
- Supports continuation strategies: `transcript`, `state`, `hybrid`, `ephemeral`.
- Extracts generated images from image generation output items.

### OpenAI VAI and Amazon Bedrock VAI providers

- Use Vercel AI SDK `generateText(...)`.
- Convert SDK messages/tools into Vercel AI SDK shapes.
- Normalize Vercel tool calls back into SDK `ToolCall[]`.
- Best-effort extract reasoning metadata/content.

### GLM provider

- Direct Z.AI chat completions integration.
- Supports `general` and `coding` endpoints.
- Maps SDK tools to OpenAI-compatible function tools.
- Supports `thinking` and `rawBody` options.
- Preserves `reasoning_content` where available.

## Storage responsibilities

Storage adapters persist normalized SDK transcripts. They should preserve:

- `role`
- `content`
- `date`
- `reasoningContent`
- `name`
- `toolCallId`
- `toolCalls`

Adapters should not persist system messages unless explicitly required by a future design. The runtime currently re-adds current agent instructions on each run.

## Testing guidance

Use focused tests for the subsystem you change:

```bash
npx vitest run tests/runtime-permissions.test.ts
npx vitest run tests/runtime-storage.test.ts
npx vitest run tests/openrouter-provider.test.ts
npm run check
npm test
```

Testing conventions:

- Use fake `ModelProvider` implementations for runtime tests.
- Use fake fetch/client/generate functions for provider tests.
- Do not make network calls in unit tests.
- For permission tests, assert both hook payloads and transcript output.
- For provider changes, assert normalized SDK outputs, not just raw provider payloads.

## Coding conventions

- TypeScript ESM with `moduleResolution: "NodeNext"`; relative imports include `.js` extensions.
- `strict`, `exactOptionalPropertyTypes`, and `noUncheckedIndexedAccess` are enabled.
- Avoid explicit `undefined` assignment for optional properties; prefer conditional spreads.
- Keep runtime/provider/storage boundaries clean.
- Prefer additive public API changes.
- Update README and tests when changing public types or behavior.
- Run `npm run check` and relevant tests before reporting completion.
- Run `npm run build` before packaging.
